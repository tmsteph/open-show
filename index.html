Any computer becomes front-of-house.

OpenShow is an open-source, network-first control stack for audio, video, and lighting. Start today with a laptop + common USB interfaces. Grow into a future where speakers, projectors, and fixtures take network in and speak open protocols — no proprietary islands.

What we’re building first

Corporate MVP
  • MVP::PLAYBACK
    Play PowerPoint and videos reliably, with a simple cue list, hotkeys, and notes. (Think: show-ready playback that doesn’t require “the big switcher.”)
  • MVP::QLAB
    Ship a QLab-class replacement for corporate playback: cue stacks, GO/BACK, and safe, repeatable show control.
  • MVP::OUTPUTS
    Output to projectors/displays using common adapters (HDMI/USB-C), and support common capture/output cards as needed.
  • MVP::EXTEND
    Add USB lighting + USB audio interfaces next, then graduate to fully networked endpoints.

The problem

Corporate events often require separate boxes for playback, switching, audio mixing, and lighting control — plus vendor-specific software that only runs on one OS. The result: heavy kits, complex workflows, fragile handoffs, and expensive gear duplication. OpenShow aims to unify control into a portable, repeatable, open system.

Too many islands

Audio, video, and lighting each have their own control worlds — and they rarely agree on standards.

OpenShow provides one showfile, one cue engine, many adapters.

OS lock-in

Control apps are often Windows-only, forcing extra laptops and extra failure points.

OpenShow is browser-first + headless engine: OS becomes a detail.

Hard to scale

When you need endpoints near stage or amps, “one laptop does it all” becomes fragile.

OpenShow uses edge nodes near outputs — FOH can be any client.

MVP: Playback that replaces the “presentation stack”

The first deliverable is intentionally narrow and gig-realistic: QLab-class PowerPoint + video playback with a cue list and reliable output routing. This is the fastest way to ship something useful for corporate events while building the foundation for audio + lighting control.

MVP features

• QLab-style cue list: PPT slides, videos, images, black/freeze
• Hotkeys + “GO / BACK” workflow
• Notes per cue + run-of-show view
• Output selection (projector / confidence / operator)
• Health page (FPS, dropped frames, device status)
• Simple “panic” actions (blackout, mute, stop)

The MVP is designed to be used at shows immediately — even before audio/lighting are integrated.

Typical corporate kit

• 1 FOH laptop (browser UI)
• 1–2 USB video output / capture options (as needed)
• Optional: tiny stage mini-PC near projector/switcher rack
• Network switch + a clean control VLAN (recommended)

Goal: walk in lighter, set up faster, and keep the system recoverable if a laptop dies.
MVP scope boundaries (what we are NOT doing yet)
• Not replacing a high-end live audio console on day one.
• Not doing full broadcast-grade switching immediately.
• Not transporting audio over the same system (Dante/AES67) in v1 — this is a control + playback foundation first.
• Not relying on proprietary cloud services to function at a gig (offline-first mindset).

Architecture: FOH clients + edge nodes

OpenShow is built like a modern control system: a headless engine coordinates devices and cues, while any computer can be the operator UI. Mini-PCs near stage outputs run adapters for USB/serial/network gear. Over time, those adapters become fully network-native endpoints.

Control plane (headless engine)

• Device discovery (find nodes + gear automatically)
• Unified API (one model for many protocols)
• Cue engine (timestamped “apply at T”)
• Permissions + audit log
• Redundancy ready (leader election / failover)

Runs on a dedicated mini-PC, or on any machine. UI clients connect via LAN.

Edge nodes (near outputs)

• USB audio interfaces (initially)
• USB DMX / lighting adapters (initially)
• Projector control (PJLink / RS-232 / HTTP)
• Local health monitoring and safe fallbacks
• Protocol bridges (OSC, MIDI, sACN, Art-Net)

If FOH Wi-Fi hiccups, edge nodes keep last-known state and can fall back to safe scenes.
FOH Laptop (Browser UI) Backup Laptop (Browser UI) \ / \ / -----> Control Plane <----- (Headless Engine) | | | | --------- --------- | | Edge Node A Edge Node B (near projectors) (near amps / lighting) - video out/control - USB audio - PJLink/RS-232 - USB DMX - status telemetry - sACN/Art-Net out
“Network-first endpoints” vision (future hardware)
Long-term, OpenShow becomes its own ecosystem of open devices:

• Speakers/amps with network-in + open telemetry
• Projectors/displays that expose open control + diagnostics
• Fixtures/nodes that are standards-compliant and self-identifying
• Firmware and schematics that are open-source, repairable, and auditable

The near-term bridge is common USB + Ethernet adapters — the long-term goal is “plug it into the switch and it shows up.”

Roadmap

We’re building in phases that each produce real-world value for live events — especially corporate — while steadily moving toward a fully network-native, open hardware ecosystem.

Phase 1 — Corporate Playback MVP

Browser UI + cue list. PowerPoint + video playback. Multi-output. Reliable hotkeys. Run-of-show and notes.

This replaces the “presentation stack” first.

Phase 2 — Lighting Adapter Layer

USB DMX support + sACN/Art-Net output. Scenes/looks. Basic busking controls. Cue integration.

This replaces entry-level lighting control workflows.

Phase 3 — Audio Control + Monitoring

USB audio routing and monitoring; integrate common control protocols (OSC/MIDI) and device telemetry.

Control + visibility first; deep console replacement later (if ever).

Phase 4 — Edge Nodes + Redundancy

Harden the node agent, leader election, offline-first behaviors, automatic recovery, and support bundles.

“Any computer can be FOH” becomes real.

Phase 5 — Network-native endpoints

Open speakers / projectors / nodes: network-in, open diagnostics, OTA updates with rollback, repairable design.

The hardware becomes simple; the network becomes the backplane.

Phase 6 — Open ecosystem

Published protocols + SDK. Community drivers. Integrations. A marketplace of open modules.

Shift the industry by making openness the easiest option.

Get involved

OpenShow is meant to be built in public. If you’re a systems tech, programmer, stagehand, designer, or manufacturer who wants open standards to win, there’s a place for you here.

What we need first

• A clean product spec for the playback MVP
• A showfile format (JSON) that can be diffed and versioned
• A minimal node agent and discovery mechanism
• UI screens: Cue List, Outputs, Notes, System Health

If it can run a corporate show without drama, it’s worth shipping.

Guiding principles

• Open-source by default
• Works offline (no cloud dependency at gigs)
• OS-agnostic UI (browser-first)
• Edge nodes near outputs for reliability
• Logs + diagnostics built-in
• Repairable hardware as the endgame

The goal is “less gear, more certainty.”

Project one-liner

OpenShow turns any laptop into FOH, uses small edge nodes near stage outputs, starts with common USB interfaces, and evolves into an open ecosystem of network-native speakers and projectors — so live production can finally be portable, reliable, and free.
